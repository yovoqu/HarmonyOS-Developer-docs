## 一、整体经验结论

这类站点抓取的关键结论有 8 条：

1. 华为开发者文档中心不适合优先用 `curl` 硬拉页面。
2. 对动态渲染文档，`Playwright` 比裸 HTTP 抓取稳定得多。
3. 批量抓取时，要按“单页隔离”设计执行单元。
4. 页面可用性判断不能只依赖固定事件，要判断正文是否真正出现。
5. 后处理必须包括清理临时文件，只保留最终产物。
6. 目录结构必须以官网实际栏目为准，不能只按口头假设直接落盘。
7. landing page 本身有归档价值，应该单独保留。
8. 官网链接列表里可能混入异常 slug 或 404 页面，必须在保存前做校验。

## 二、为什么一开始的方案不稳定

### 1. `curl` 直连页面不稳定

最开始尝试直接抓：

- 页面 URL
- 文档接口 URL

遇到的典型问题：

- 连接重置
- 超时
- 本地代理环境变量干扰
- 页面返回的只是壳，不包含最终正文

原因本质上是：

- 官网有动态渲染
- 页面依赖前端 JS 请求正文
- 文档接口本身也不一定适合脱离浏览器上下文直接访问

结论：

- 这种站点不能把 `curl` 作为主方案
- `curl` 最多适合做探测，不适合做正文抓取主路径

### 2. 裸接口抓取也不适合作为主路径

虽然通过浏览器网络监听能看到官方文档接口，比如：

- `getDocumentById`
- `getCatalogTree`
- `getNavigationAddress`

但直接复刻请求时仍然可能遇到：

- connect timeout
- connection reset
- 请求上下文不完整
- 反爬或链路策略问题

这意味着：

- “看到接口”不等于“适合大规模直接调用接口”

最稳的方式还是：

- 让浏览器正常打开页面
- 等页面真实渲染出正文
- 再从 DOM 中提取内容

## 三、最有效的抓取方案

### 1. 用 Playwright 打开页面

最终采用的稳定方案是：

1. 用 `Playwright` 打开文档页
2. 等待页面至少完成初步提交
3. 轮询正文区域是否已有实际内容
4. 从 `.markdown-body` 提取正文 HTML
5. 保存为本地 HTML 文件

相比 `curl`，这个方法的优势是：

- 更接近真实用户访问
- 能拿到最终渲染后的内容
- 能绕开很多接口层面的不稳定问题

### 2. 先固定 URL 清单，再抓取

这一步非常关键。

不要一开始就无限递归爬取所有页面，而应该先从几个总览页中整理出固定文档清单。

本次实际做法是：

1. 先打开几个总览或入口页
2. 提取页面中所有 `ui-design-*` 链接
3. 把目标文档 URL 固化成清单
4. 再对清单逐篇抓取

这样做的好处：

- 范围可控
- 不会误抓无关文档
- 出错时容易补抓
- 可以明确知道“应抓多少篇”

后续在抓取 `Ability Kit` 和 `Accessibility Kit` 时，这条经验又被验证了一次：

- `Ability Kit` 的主入口页确实存在多个子栏目
- 但 `Accessibility Kit` 实际只有 `ArkTS API` 和 `错误码`
- 如果一开始不先核对官网栏目，而是直接按想象建目录，就会建错层级

结论：

- 口头目录结构只能作为初始假设
- 最终目录结构必须以官网主目录页和子目录页为准

### 3. 每个 URL 必须独立抓取

这是这次最重要的经验之一。

早期做法是：

- 一个浏览器 page 连续抓很多篇文档

问题在于，一旦某一页出现：

- `goto` 超时
- 页面卡住
- context 关闭
- 页面异常跳转

后面一串文档都会跟着失败。

后来的改进是：

- 每个 URL 独立开一个页面
- 抓完就关闭
- 每一篇文档成为独立执行单元

这样带来的好处：

- 单页失败不会拖死整个批次
- 日志更清楚
- 更容易补抓缺失项

另外还有一个很实用的补充：

- 不同 Kit 最好使用独立脚本

原因是：

- 每个 Kit 的栏目结构不同
- 有的只有两层
- 有的有三级目录
- 有的适合递归
- 有的更适合固定清单

例如本次：

- `UI Design Kit` 用固定 URL 清单更稳
- `Ability Kit` 适合按已知栏目递归抓取
- `Accessibility Kit` 则适合按官网实际少量页面直接固定抓取

结论：

- 不要强求所有模块共用一套完全一致的抓取策略
- 最稳妥的是“一类模块一份脚本设计”

## 四、关键步骤拆解

## 1. 先确认文档归类规则

真正开始抓取前，必须先把分类规则说清楚。

本次规则很明确：

- 4 篇指定文档进入 `ArkTS API`
- 其余 `ui-design-*` 文档进入 `ArkTS组件`

如果这一步不固定，后面容易出现：

- 已抓完但目录不对
- 后续还要移动文件
- 清单和目录不一致

结论：

- 先定规则，再抓内容

## 2. 先建立最终目录，不要先建临时目录

本次直接建立：

- `huawei_dev_docs/应用框架/ArkTS API`
- `huawei_dev_docs/应用框架/ArkTS组件`

这么做的好处是：

- 每篇文档抓到后可以立即放进最终位置
- 不需要最后再做二次搬运
- 后续生成 `manifest.json` 更直接

## 3. 从总览页拿完整列表

这一步的重点不是“抓正文”，而是“先得到完整文档列表”。

操作方式：

1. 打开总览页
2. 抽取所有链接
3. 过滤出包含 `ui-design-` 的 URL
4. 去重
5. 再由人工确认哪些属于这次范围

本次得到的来源页主要包括：

- references 里的总览页
- guides 里的总览页

扩展到后续模块时，这一步还应该再细化成两级：

1. 先拿主目录页的栏目入口
2. 再分别进入每个栏目页，抽正文页链接

这样做比“一次性从主目录页抽所有链接”更可靠，因为：

- 主目录页只负责给出一级导航
- 真正文档往往挂在二级栏目页里
- 不同栏目页的链接质量和覆盖范围不同

这一步的价值非常大，因为后面所有动作都建立在“列表准确”之上。

## 4. 单页抓取的等待策略

这是抓取成功率的核心细节。

早期不稳定的等待方式是：

- `waitUntil: "domcontentloaded"`
- 然后直接等 `.markdown-body` visible

问题是：

- 有些页面 `domcontentloaded` 非常慢
- 有些页面 DOM 已建立，但正文还没注入
- 有些页面会延迟渲染

后来的稳定策略是：

1. `goto` 只等到较早阶段，比如 `commit`
2. 然后轮询页面状态
3. 判断 `.doc-title` 或 `.markdown-body` 是否已有实际内容

这个逻辑比“死等某个固定事件”更稳。

建议判断条件：

- 标题是否非空
- 正文容器是否存在
- 正文容器 `innerHTML` 是否非空

## 5. 批量抓取要分两阶段

推荐流程：

### 第一阶段：主批次抓取

- 直接对完整 URL 清单批量抓
- 记录成功项
- 记录失败项

### 第二阶段：失败项补抓

- 从失败日志中提取缺失 URL
- 单独重跑这些 URL
- 可并行补抓

本次实践证明：

- 一轮抓完不是重点
- 可恢复、可补抓才是重点

如果模块本身页数不大，也可以直接采用：

- landing page 固定抓
- 正文页固定抓

这在 `Accessibility Kit` 这类小模块上更简单，也更不容易误抓。

## 6. 补抓一定按 URL 粒度并行

这是本次后半段最有效的提速方式。

前面尝试过“把很多 URL 交给同一个进程处理”，效果不好，因为：

- 某一页卡住会拖慢整批
- 后续错误传播严重

后来的方式是：

- 先整理缺失 URL 列表
- 再用并行方式逐 URL 处理

这种方式的优点：

- 某一篇失败，不影响其余篇
- 并行效率高
- 非常适合补缺

结论：

- 批量抓取适合分组
- 补抓适合逐 URL 并行

## 五、抓取过程中遇到的典型问题

### 1. 连接相关问题

- `curl` 连接被重置
- 接口 connect timeout
- 本地代理变量干扰请求路径

处理思路：

- 不再强依赖 `curl`
- 转用浏览器访问链路

### 2. 页面加载相关问题

- `page.goto` 超时
- 页面框架已打开，但正文没出现
- `.markdown-body` 一直没法按时 visible

处理思路：

- 降低对单一事件的依赖
- 使用更宽松的进入策略
- 加入轮询正文内容

### 3. 上下文污染问题

如果同一个 `page` 连续跑很多文档，会遇到：

- 一页失败导致 page 状态异常
- 后续页面跟着失败
- 报错变成 `Target page, context or browser has been closed`

处理思路：

- 一页一个 page
- 抓完立即关闭

### 4. 范围膨胀问题

如果一开始不锁定 URL 范围，而是递归追所有页面链接，很容易：

- 抓取时间失控
- 抓到不在本次范围内的页面
- 难以确认是否真正完成

处理思路：

- 先固化清单
- 再执行抓取

## 六、HTML 文档格式规范

这部分非常重要，因为最终产物不是原始 JSON，而是本地可读 HTML。

本次生成的 HTML 文档遵循以下规范。

### 1. 文档结构

每篇文档都使用统一外壳，包含：

- `<!doctype html>`
- `<html lang="zh-CN">`
- `<head>`
- `<meta charset="utf-8">`
- `<meta name="viewport" content="width=device-width, initial-scale=1">`
- `<title>`
- 内联 `<style>`
- `<body>`

页面结构采用统一容器：

- 外层 `.page`
- 头部 `.meta`
- 正文 `.markdown-body`

这样做的目的是：

- 统一离线阅读体验
- 避免依赖官网 CSS
- 后续更容易转 Markdown 或再加工

从后续 `Ability Kit` 和 `Accessibility Kit` 的实践看，还应再补一条：

- landing page 和正文页应当使用同一套 HTML 外壳

这样做的好处是：

- 目录页和正文页视觉风格一致
- 后续检索和转换脚本无需区分两套格式
- manifest 中只需要通过 `kind` 字段区分 `landing` 和 `doc`

### 2. 元信息规范

每篇文档顶部保留：

- 文档标题
- 原始来源 URL

来源 URL 必须保留，原因是：

- 方便回溯官网原文
- 便于后续校对更新
- 方便知识库内引用

### 3. 样式规范

为了保证离线可读性，样式上做了这些约束：

- 白底卡片式布局
- 固定最大宽度
- 合理留白
- 默认中文无衬线字体
- 代码块单独深色背景
- 表格强制边框和横向滚动
- 图片 `max-width: 100%`

目标不是完全复刻官网样式，而是：

- 保证可读
- 保证结构清晰
- 保证在脱离外部资源时依然能看

### 4. 内容规范

正文主体来自官网页面中的：

- `.markdown-body`

也就是说，本地 HTML 保留的是：

- 渲染后的正文 HTML

而不是：

- 整个官网页面壳
- 导航栏
- 页脚
- 登录栏
- 侧边站点公共布局

这个选择的好处：

- 文件更小
- 内容更纯净
- 更适合归档和二次处理

### 5. 文件命名规范

本次统一使用页面 slug 命名：

- `ui-design-symbolregister.html`
- `ui-design-navigation.html`

优点：

- 与官网 URL 一一对应
- 易检索
- 易生成清单
- 易做补抓比对

但实践中要注意一个例外：

- 官网偶尔会出现异常 slug
- 例如链接前缀缺字母、或者生成出的 URL 本身不规范

这时不能机械照搬，而应该：

1. 先验证该链接是否真能打开正文
2. 如果是官网异常但页面有效，本地文件名可按同组命名规则做修正
3. 如果页面实际是 `404`，则不要保留

例如本次：

- `Ability Kit` 中出现了一个异常 slug，本地已按同组规则修正
- `Accessibility Kit` 中有一条 `AccessibilityExtensionContext` 链接实际返回 `404`，最终没有保留

### 6. 目录规范

目录层级尽量简单，只保留：

- 分类目录
- HTML 文件
- 一个 `manifest.json`

这样最适合后续：

- 手动浏览
- 程序读取
- 导入知识库

## 七、manifest 的作用

`manifest.json` 是整个抓取任务里非常关键的辅助文件。

它至少应包含：

- `group`
- `category`
- `slug`
- `kind`
- `file`
- `url`

它的价值在于：

- 知道最终抓到了哪些文档
- 能快速反查来源链接
- 后续可以据此做增量更新
- 后续可以用它批量转 Markdown

补充说明：

- `group` 适合表示实际目录分组，例如 `Stage模型能力的接口`
- `kind` 适合标记这是 `landing` 还是 `doc`

这两个字段在后续抓取 `Ability Kit` 这类多级目录模块时非常重要。

建议以后继续保留这个文件。

## 八、建议的标准操作流程

以后如果再做类似任务，建议直接按这个顺序：

1. 明确抓取范围
2. 明确目录分类规则
3. 创建最终目录
4. 从总览页提取固定 URL 清单
5. 去重并人工确认范围
6. 确认官网实际栏目层级是否与预期一致
7. 决定本次模块更适合“固定清单”还是“按栏目递归”
8. 用 Playwright 单页抓正文
9. landing page 与正文页都保存为统一格式 HTML
10. 生成 `manifest.json`
11. 过滤 404 页面和异常链接
12. 统计缺失项
13. 对缺失项逐 URL 并行补抓
14. 复核数量、目录分类和文件命名
15. 删除临时文件

## 九、哪些文件应该清理

任务完成后，建议删除：

- 临时抓取脚本
- 调试输出
- 缺失 URL 列表
- 分片清单

建议保留：

- 最终 HTML 文档
- `manifest.json`

这么做的好处：

- 目录干净
- 可交付
- 后续维护成本低

## 十、最值得记住的几点

如果只记 6 条，记下面这些就够了：

1. 先锁定文档清单，不要无限递归抓。
2. 动态文档优先用 Playwright，不要先赌 `curl`。
3. 一页一个 page，不要一页失败拖垮整批。
4. 判断“正文是否存在”比依赖固定加载事件更稳。
5. 补抓阶段按 URL 粒度并行，效率最高。
6. 官网目录结构和链接质量必须实测，不要完全相信口头结构或页面里给出的每一个链接。

## 十一、后续新增经验

这一轮继续抓取 `Ability Kit` 和 `Accessibility Kit` 后，又新增了几条很值得保留的经验：

### 1. 不同 Kit 的结构复杂度差异很大

例如：

- `UI Design Kit` 更适合固定清单抓取
- `Ability Kit` 有明显的多级栏目结构
- `Accessibility Kit` 实际页面量很小

结论：

- 抓取方案应该跟着模块结构走
- 不要用一套过于僵硬的统一方案去套所有模块

### 2. landing page 处理规则

- **HTML 归档阶段**：保留 landing page 为 `*.landing.html`（有价值：模块概览、子栏目入口、分类说明）
- **Markdown 产出阶段**：不保留 landing page。因为 navigation page 正文价值低于具体文档，会污染 Obsidian 检索结果。用户有目录结构即可

### 3. 官网存在坏链接或异常链接

这次实际遇到了两类情况：

- 官网给出的链接能解析，但 slug 不规范
- 官网给出的链接实际打开后是 `404`

处理原则：

- 页面有效：修正本地命名
- 页面无效：从产物和清单中剔除

### 4. manifest 不只是索引，也是校验工具

在多级目录模块中，`manifest.json` 不仅用来记录文档，还可以用来：

- 统计每个分组的数量
- 找出异常 slug
- 找出 404 页面
- 验证目录结构是否符合预期

所以建议每次抓取结束后都做一次：

- 数量统计
- 异常项检查
- 文件路径抽样检查

## 十二、大规模批量抓取新增经验（2026-05）

这一轮完成了开发/指南下全部 4330 篇文档的批量抓取，新增以下经验。

### 1. 优先使用 getCatalogTree API 获取完整目录树

华为开发者文档站提供了内部 API，可以直接获取完整的导航目录树，无需从侧边栏 DOM 逐层提取：

- **API 端点**：`POST /community/servlet/consumer/cn/documentPortal/getCatalogTree`
- **请求体**：`{"language":"cn","catalogName":"harmonyos-guides","objectId":"application-dev-guide"}`
- **返回数据**：完整的树形 JSON，每个节点包含 `nodeName`、`relateDocument`、`isLeaf`、`children`、`relateDocId`

相比从侧边栏 DOM 提取，使用 API 的优势：

- 结构准确，无需处理 DOM 嵌套层级
- 直接获取全树而非可见部分
- 叶子节点标记明确（`isLeaf: true`），易于过滤

### 2. getDocumentById API 可以高效获取正文

通过 `getDocumentById` API，传入文档 slug 即可获取完整的文档 HTML：

- **API 端点**：`POST /community/servlet/consumer/cn/documentPortal/getDocumentById`
- **请求体**：`{"objectId":"application-dev-guide","version":"","catalogName":"harmonyos-guides","language":"cn"}`
- **返回数据**：包含 `title`、`displayUpdateTime`、`content.content`（HTML正文）、`anchorList` 等

优势：

- 绕过动态渲染问题，无需等页面加载完成
- 数据干净，直接是正文 HTML（不含导航/页脚）
- 包含准确的更新时间

### 3. 目录树中的中间层级需要清洗

官网目录树中，某些一级节点下会插入额外的中间层级（如"开发"、"指南"），这些中间层在本地目录结构中没有意义。例如：

- **问题路径**：`开发/指南/开发/AI/开发/Agent Framework Kit/...`
- **正确路径**：`开发/指南/开发/AI/Agent Framework Kit/...`

处理方式：

- 遍历目录树构建本地路径时，过滤掉 sectionNode 自身名称
- 同时过滤掉中间层中与父级重复或冗余的通用目录名（如"开发"）

### 4. 用户分类不一定对齐官网分类

用户期望的分类（入门/开发/工具/测试/应用体验）可能与官网一级节点不完全对应。需要二次确认映射关系。例如：

- "工具"应包含：开发环境搭建、使用AI智能辅助编程、编写与调试应用、构建应用、优化应用性能、发布应用、命令行工具
- 其中部分节点在官网树中与"开发"类节点混在同一层级

结论：

- 抓取前先输出完整一级节点列表让用户确认分类映射
- 不要假设官网的一级节点 = 用户的分类目录

### 5. Markdown 产出后需要三步格式修复

批量抓取后，由于 HTML 转换的固有局限，必须对全量 Markdown 文件执行三项格式修复：

1. **图片与文本分行**：转换后的 `![](assets/...)` 常与段落文字挤在同一行，导致 Obsidian 中图片无法正常渲染。必须将图片引用拆分为独立行。
2. **行首空格清除**：HTML 标签转换后残留大量行首空白，影响排版一致性。
3. **段落内多余空格压缩**：HTML 中的 `&nbsp;` 和 DOM 缩进在转换后变成大量连续空格，需用单空格替换。

这些修复应在抓取完成后作为自动化后处理步骤执行。

### 6. 大规模抓取需要断点续传

4000+ 篇文档的抓取耗时较长（数十分钟），应设计检查点机制：

- 每个 section 完成后立刻写入 manifest.json
- 重新运行时读取已有 manifest，跳过已完成文档
- manifest 同时作为进度记录和最终产出物

### 7. API 参考文档的设备支持信息必须提取（2026-05 新增）

华为 API 参考文档（harmonyos-references）的 HTML 中，`<h1>` 标签带有 `device-type` 属性，记录了该接口支持的设备类型：

```html
<h1 device-type="phone,2in1,tablet,wearable,tv">@ohos.arkui.inspector (布局回调)</h1>
```

转换 Markdown 时必须提取此属性，映射为可读的设备列表并写入文档头部：

```md
支持设备：Phone | PC/2in1 | Tablet | TV | Wearable
```

关键经验：
- 只有 `harmonyos-references` 类型的文档才有此属性，guides、best-practices、faqs、releases 均无
- 不是所有 references 文档都有 device-type（如概述类页面 `development-intro-api` 无）
- 不同接口支持的设备集合不同，不能使用固定字符串
- 设备代码映射：phone→Phone, 2in1→PC/2in1, tablet→Tablet, wearable→Wearable, tv→TV

### 8. API 元数据项需要空行分隔（2026-05 新增）

API 参考文档中，每个接口的 `**元服务API：**`、`**系统能力：**`、`**模型约束：**`、`**参数：**` 等元数据行在 HTML 转换后常常紧贴在一起，在 Obsidian 中可读性很差。

必须在后处理阶段将连续的 `**xxx：**` 行用空行分隔，并在 `**参数：**` / `**返回值：**` 与紧随的表格之间插入空行。

结论：
- HTML 转 Markdown 后需要专门的 API 文档元数据间距处理
- 这条规则应在 Markdown 格式修复步骤中与其他后处理一起执行
