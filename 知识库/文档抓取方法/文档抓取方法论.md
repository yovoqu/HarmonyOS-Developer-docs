# 文档抓取方法论

## 目的

这篇文档只讲“怎么抓文档”这件事本身，不限定某个具体站点。

适用场景：

- 官方文档站
- 前端动态渲染页面
- 需要离线归档的知识文档
- 需要分类保存的 API / 指南 / 错误码文档

## 一、先做范围确认，不要直接开爬

开始前先确认 4 件事：

1. 抓什么范围
2. 怎么分类
3. 最终目录长什么样
4. landing page 是否需要保留

如果这一步没做，后面最常见的问题就是：

- 抓多了
- 抓偏了
- 分类不一致
- 产物需要返工搬目录

## 二、先拿导航，再拿正文

正确顺序通常是：

1. 先打开主目录页
2. 提取一级入口
3. 再进入二级栏目页
4. 最后才抓正文页

不要一开始就无限递归页面里的所有链接，因为这样很容易：

- 抓到无关文档
- 抓到全站公共链接
- 抓取规模失控

## 三、优先浏览器抓取，不优先裸请求

对于现代文档站，优先级一般是：

1. `Playwright` 这类真实浏览器抓取
2. 必要时再研究接口
3. `curl` 只做探测，不做主路径

原因：

- 页面常常动态渲染
- 接口可能依赖上下文
- 页面里的正文未必直接在初始 HTML 中

## 四、页面可用性的判断要看“正文是否存在”

不要只依赖：

- `domcontentloaded`
- `networkidle`
- 某个固定 selector 是否出现

更稳妥的做法是轮询：

- `.doc-title` 是否有值
- `.markdown-body` 是否有内容
- 正文容器的 `innerHTML` 是否非空

关键不是“页面加载事件是否触发”，而是“正文是否真的出来了”。

## 五、执行单元要足够小

批量抓取时，最稳的执行粒度通常是：

- 一页一个 `page`
- 一篇一个任务

这样可以避免：

- 某一页超时拖垮整批
- 页面状态污染后续抓取
- context 关闭导致一串失败

如果页数较多，可以：

- 按栏目分批
- 对失败项逐 URL 并行补抓

## 六、固定清单和递归抓取要分场景

### 适合固定清单的场景

- 页面数量少
- 栏目结构清晰
- 目标文档列表容易人工确认

优点：

- 更可控
- 不容易误抓

### 适合按栏目递归的场景

- 主目录页只给栏目入口
- 正文页集中挂在某个栏目 landing page 下
- 文档量较大

优点：

- 减少手工维护 URL 列表

结论：

- 不要执着于统一方案
- 根据模块复杂度选“固定清单”或“栏目递归”

## 七、landing page 要不要保留

通常应该保留。

因为 landing page 往往包含：

- 模块概览
- 子栏目导航
- 目录结构说明

建议单独标记为：

- `*.landing.html`

这样后续能明确区分：

- landing
- 正文页

## 八、HTML 产物要统一格式

本地离线文档最好统一外壳，至少包含：

- `doctype`
- `title`
- 来源 URL
- 正文容器
- 内联基础样式

统一格式的价值：

- 易读
- 易检索
- 易转 Markdown
- 易批处理

## 九、manifest 必须保留

建议每次抓取都输出 `manifest.json`。

推荐字段：

- `group`
- `slug`
- `kind`
- `file`
- `url`
- `title`

它的作用不只是索引，还包括：

- 数量统计
- 缺失校验
- 异常项排查
- 后续增量更新

## 十、抓取结束后的检查动作

至少要检查：

1. 目录数量是否符合预期
2. 文件命名是否统一
3. manifest 是否可用
4. 是否混入 404 页面
5. 是否存在异常 slug

## 十一、清理原则

任务完成后，一般只保留：

- 最终 HTML
- `manifest.json`

其余可以清理：

- 临时脚本
- 调试输出
- 分片清单
- 缺失 URL 列表

## 十二、最简工作流

1. 明确范围和分类
2. 创建最终目录
3. 先拿 landing page 和栏目入口
4. 决定固定清单还是栏目递归
5. 用浏览器抓正文
6. 统一保存为 HTML
7. 生成 `manifest.json`
8. 校验 404 / 异常 slug
9. 补抓缺失项
10. 清理中间文件

## 十三、大规模批量抓取特殊考虑

当文档量达到千级以上时，通用工作流需要补充以下策略。

### 1. 优先使用内部 API 而非 DOM 解析

小规模抓取适合从页面 DOM 提取导航，但大规模抓取应优先寻找和利用站点内部 API：

- 导航目录 API（如 `getCatalogTree`）直接返回完整 JSON 树，比 DOM 提取更准确
- 文档正文 API（如 `getDocumentById`）直接返回纯净 HTML 正文，绕过动态渲染
- 认证方式：先通过 Playwright 打开任意页面建立浏览器会话，后续 API 调用复用 Cookie

### 2. 用户分类与官网分类需要二次确认

大规模抓取前，先输出完整的一级/二级节点列表让用户确认分类映射。官网分类（如"应用框架"、"系统"）不一定对应最终目录分类（如"入门"、"开发"、"工具"）。抓取后再移动文件成本很高。

### 3. 目录树中的冗余层级需要清洗

官网导航树中可能插入只起分组作用的中间层级（如"开发"、"指南"），这些层级的名称在本地目录中无意义。构建本地路径时应过滤掉与父级重复的冗余层级名。

### 4. Markdown 产出后需要自动化格式修复

HTML 转 Markdown 后会残留格式问题，必须作为独立后处理步骤自动化执行：

- 图片引用与文本分行（防止图片无法渲染）
- 行首空格清除（防止意外触发代码块）
- 段落内多余空格压缩（提升可读性）
- 特殊符号转义（如 `$` 被 Obsidian 解析为 LaTeX）

### 5. 断点续传必须内置

千级以上文档抓取耗时较长，必须设计检查点：
- 每个批次完成后即时写入 manifest
- 重跑时读取已有 manifest 跳过已完成项
- 不要依赖外部进度记录
