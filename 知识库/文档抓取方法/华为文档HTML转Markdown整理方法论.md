# 华为文档HTML转Markdown整理方法论

## 相关阅读

- [[文档抓取方法论]]
- [[华为开发者文档抓取实战规范]]
- [[华为开发者文档拉取经验总结]]

## 目的

这篇文档不讲“怎么把官网页面抓下来”，而是专门讲：

- 已经抓到本地的 HTML，如何整理成可长期维护的 Markdown
- 如何适配 Obsidian
- 如何保证图片、表格、代码块、更新时间、文件命名都稳定
- 在华为开发者文档场景下，哪些规则必须固定下来

这套方法当前已经实战用于：

- `huawei_dev_docs/指南/应用框架/...` 中的离线 HTML
- `huawei_docs_md/指南/应用框架/...` 中的 Markdown 重建

## 一、整体目标

Markdown 整理的最终目标不是“把 HTML 粗暴转成 md”，而是：

1. 可读
2. 可检索
3. 可在 Obsidian 中长期维护
4. 目录稳定
5. 附件路径稳定
6. 代码块语言正确
7. 尽量贴近官网原始排版

如果只追求快速转 md，通常会出现这些问题：

- 文件名还是 URL slug，不是文档名
- 图片还是过期签名 URL
- 代码块混入 `收起 / 复制 / 自动换行`
- 表格残留 HTML
- landing page 和正文页混在一起
- 文档头部没有官网真实更新时间
- 同一篇文档内不同语言代码块被统一标错

## 二、目录设计原则

Markdown 目录要和 HTML 归档目录分离。

推荐结构：

```text
huawei_dev_docs/
  指南/
    应用框架/
      Background Tasks Kit/
        *.html
        manifest.json

huawei_docs_md/
  指南/
    应用框架/
      Background Tasks Kit/
        *.md
        assets/
          文档名/
            file-时间戳.png
        manifest.json
```

这样做的好处：

- HTML 归档保留原始中间产物
- Markdown 可反复重建
- 不会因为清洗 Markdown 影响原始 HTML
- 后续发现转换规则有问题时，可以直接回源重建

## 三、适用范围判断

HTML 转 Markdown 之前，先判断这批页面是不是“正文页”。

当前实践规则：

- `landing` 页默认不转 Markdown
- `doc` 页转 Markdown

原因：

- landing 页主要是导航页
- 正文价值低于具体文档
- 会污染 Obsidian 检索结果
- 用户实际需要的是具体文档内容

例外情况：

- 如果 landing 页本身包含独立说明、完整概述、且用户明确要求保留，才转

## 四、文件命名规范

### 1. 不使用 URL slug 作为 Markdown 文件名

错误做法：

- `background-task-overview.md`
- `transient-task.md`

正确做法：

- `Background Tasks Kit简介.md`
- `短时任务(ArkTS).md`
- `延迟任务(ArkTS).md`

原因：

- 文档名对人更友好
- 在 Obsidian 中检索更自然
- 能和官网标题直接对齐

### 2. 文件名优先使用官网文档名

来源优先级：

1. 官网 `h1`
2. manifest 中的 `name`
3. 最后才考虑 slug

### 3. 非法文件名字符必须替换

文件系统中不能直接使用这些字符：

- `/`
- `\`
- `:`
- `*`
- `?`
- `"`
- `<`
- `>`
- `|`

推荐替换方案：

- `/` -> `／`
- `\` -> `／`
- `:` -> `：`
- `*` -> `＊`
- `?` -> `？`
- `"` -> `＂`
- `<` -> `＜`
- `>` -> `＞`
- `|` -> `｜`

例如：

- 官网标题 `短时任务(C/C++)`
- 本地文件保存为 `短时任务(C／C++).md`

原则是：

- 最大程度保留原始标题视觉语义
- 不使用生硬下划线破坏标题

## 五、文档头部规范

每篇 Markdown 头部固定保留三部分：

```md
# 文档标题

更新时间：2026-03-12 17:39

来源：https://developer.huawei.com/consumer/cn/doc/...
```

### 1. 标题

- 使用官网 `h1`
- 不使用本地文件名反推标题

### 2. 更新时间

必须使用官网页面里的真实更新时间，不允许：

- 留空
- 手工猜测
- 使用抓取时间代替

更新时间抓取来源通常是：

- `.doc-header-footer .dhf-left`
- 或页面头部中包含 `更新时间:` 的节点

注意：

- 时间格式保留官网原样
- 一般为 `YYYY-MM-DD HH:mm`

### 3. 来源

- 保留官网正文页 URL
- 方便日后核对
- 方便回源重建

### 4. 设备支持（仅 API 参考文档）

对于 `harmonyos-references`（API 参考）类型的文档，官网 HTML 的 `<h1>` 标签上携带 `device-type` 属性，指示该接口支持的设备类型：

```html
<h1 device-type="phone,2in1,tablet,wearable,tv">@ohos.arkui.inspector (布局回调)</h1>
```

必须提取并转换为可读格式写入 Markdown 头部：

```md
支持设备：Phone | PC/2in1 | Tablet | TV | Wearable
```

设备代码与显示名称的映射：

| 代码 | 显示名称 |
| --- | --- |
| phone | Phone |
| 2in1 | PC/2in1 |
| tablet | Tablet |
| wearable | Wearable |
| tv | TV |

注意：
- 不同接口支持的设备集合不同（如某些接口仅支持 `Phone | Tablet | TV`）
- 非 API 参考文档（guides、best-practices、faqs、releases）的 h1 上无此属性，不需要添加
- `development-intro-api` 等概述类 API 页面也可能无此属性

## 六、图片处理规范

### 1. 不能直接保留官网签名 URL

华为官网图片大量使用带签名的临时 URL。

如果直接保留：

- 几小时或几天后链接会失效
- Markdown 中图片变成 404

所以正确流程是：

1. 回到官网实时页面
2. 获取当前有效的图片 URL
3. 下载为本地附件
4. Markdown 中引用本地路径

### 2. 必须按 Obsidian Custom Attachment Location 规则落盘

当前约定：

- 新附件位置：`./assets/${noteFileName}`
- 附件名：`file-${date:{momentJsFormat:'YYYYMMDDHHmmssSSS'}}`
- Markdown URL：`assets/${noteFileName}/${generatedAttachmentFileName}`

本地最终效果示例：

```text
Background Tasks Kit/
  Background Tasks Kit简介.md
  assets/
    Background Tasks Kit简介/
      file-20260414091711168.png
```

Markdown 引用：

```md
![](assets/Background Tasks Kit简介/file-20260414091711168.png)
```

### 3. 附件目录名必须与 noteFileName 一致

不要出现：

- Markdown 文件名是中文
- 附件目录名还是 slug

应保持一致，例如：

- 文档：`短时任务(ArkTS).md`
- 附件目录：`assets/短时任务(ArkTS)/`

## 七、正文提取规范

正文来源以官网实时页面 `.markdown-body` 为准。

不建议只从本地 HTML 二次抽取图片 URL，因为会遇到：

- 图片 URL 过期
- 某些动态内容已经过时

推荐流程：

1. 从本地 HTML 读取来源 URL
2. 再打开官网实时页面
3. 重新抓正文 DOM
4. 去掉无关控件
5. 再做 Markdown 转换

## 八、必须删除的无关内容

这些内容属于官网阅读交互控件，不属于文档正文，必须去掉：

- `收起`
- `自动换行`
- `深色代码主题`
- `复制`
- 代码块头部工具栏
- 展开/收起表格控件
- 锚点图标

可通过删除这些 DOM 节点实现：

- `.highlight-div-header`
- `.expand-box`
- `.anchor-icon`

原则是：

- 保留文档内容
- 删除阅读器控件

## 九、段落与基础排版规范

### 1. 标题

- `h1 -> #`
- `h2 -> ##`
- `h3 -> ###`
- `h4 -> ####`

### 2. 段落

- `p` 转为普通 Markdown 段落
- 段落之间保留空行

### 3. 列表

- `ul -> -`
- `ol -> 1. 2. 3.`
- 多层列表保持缩进

### 4. 强调

- `strong/b -> **...**`
- `em/i -> *...*`

### 5. 链接

- 转为 `[文本](URL)`
- 不要只保留裸 URL
- 官网相对链接如 `/consumer/cn/doc/...` 必须规范化为完整绝对链接：`https://developer.huawei.com/consumer/cn/doc/...`
- 不要在 Obsidian 中保留站内相对链接，因为脱离官网域名后会直接失效

### 6. 提示块

像官网中的“说明”“注意”这类块，建议转为 Obsidian/Markdown callout：

```md
> [!NOTE] 说明
> 内容...
```

## 十、表格转换规范

### 1. 表格必须转 Markdown，不保留 HTML

错误做法：

- 把 `<table>` 原样塞进 md

正确做法：

```md
| 接口名 | 描述 |
| --- | --- |
| foo | bar |
```

### 2. 单元格内容处理规则

- 单元格里的链接仍然保留为 Markdown 链接
- 单元格里的换行可转成 `<br>`
- 单元格中的 `|` 必须转义为 `\|`

### 3. 表格宽度不一致时补空列

转换时应先统计最大列数，再补齐，避免 Markdown 表格断裂。

## 十一、代码块转换规范

这是最容易出错的部分。

### 1. 必须使用标准 fenced code block

正确格式：

````md
```ArkTS
@Entry
@Component
struct Demo {}
```
````

不要再使用：

- `'''ArkTS`
- 伪代码块
- 缺少结束 fence 的写法

### 2. 代码块内容只能取 `pre` 里的真实代码

很多官网代码块外层有：

- 工具栏
- 行号
- 示例链接

因此必须：

1. 先删代码块头部工具栏
2. 只取 `pre`
3. 如果 `pre` 内有 `ol.linenums`
4. 就按每个 `li` 还原每一行代码

这样才能避免：

- 混入按钮文字
- 行号污染代码
- 丢掉换行

### 3. 代码块语言不能一刀切

这是实践中最关键的一条。

不能所有代码块都写成 `ArkTS`。

必须按代码块真实来源文件判断：

#### `.ets`

- 标记为 `ArkTS`

#### `.ts` / `.d.ts`

- 标记为 `ts`

#### `.cpp` / `.cc` / `.cxx` / `.h` / `.hpp`

- 标记为 `cpp`

#### 日志输出 / CMakeLists / 纯文本示例

- 标记为 `text`

### 3.1 代码块内部必须整理缩进

这条规则不能省略。哪怕代码内容已经抓到了，如果缩进没有恢复，Obsidian 里的可读性仍然会很差。

最低要求：

- 同一代码块内必须使用统一缩进风格
- 当前统一采用 4 个空格作为层级缩进
- 代码块内部不能保留网页抽取导致的“整块左对齐”状态

例如错误情况：

````md
```ArkTS
let obj = {
'a': 1,
'b': 2
}
```
````

正确情况：

````md
```ArkTS
let obj = {
    'a': 1,
    'b': 2
}
```
````

### 3.2 结构化配置块必须格式化

对于这类内容：

- JSON
- JSON5
- 结构化对象字面量
- 配置列表示例

不能仅仅“保留原始换行”，必须整理为标准层级结构。

例如：

````md
```text
{
    "UniformDataTypeDeclarations": [
        {
            "TypeId": "com.example.mySecondHap.image",
            "BelongingToTypes": [
                "com.example.myFirstHap.image"
            ],
            "FilenameExtensions": [
                ".myImageEx",
                ".khImageEx"
            ],
            "MIMETypes": [
                "application/my-ImageEx",
                "application/khImageEx"
            ],
            "Description": "My Image extension.",
            "ReferenceURL": ""
        }
    ]
}
```
````

处理原则：

- 花括号、方括号层级必须清晰
- 数组项换行时要继续缩进
- 不能出现第一行有缩进、第二行没缩进的断裂格式

### 3.3 续行表达式要补齐续行缩进

网页源码示例中经常出现这种断行：

```ts
let record =
new Something(...)
```

Markdown 中必须整理为：

```ts
let record =
    new Something(...)
```

否则阅读时会误判为两个独立语句。

### 3.4 命令行与日志块不做“代码美化式重排”

以下内容虽然通常也标记为 `text`，但不能按编程语言缩进去“重写”：

- shell 命令行会话
- hdc / bm / aa / sqlite / vector 工具输出
- 日志输出
- ASCII 目录树
- 数据库表格输出

这类块的原则是：

- 保留原始命令顺序
- 保留原始提示符
- 保留原始输出布局
- 只清理明显的网页噪声，不做语义改写

### 3.5 链式调用和声明式 UI 代码按源码阅读习惯整理

对于 ArkTS / TS 中常见的：

- 链式调用
- Fluent API
- 声明式 UI DSL

不要强行机械缩进成另一种风格，而是：

- 先恢复基本层级
- 保持链式调用的连续可读
- 尽量贴近官网示例源码的阅读习惯

判断标准不是“格式化器是否会改动”，而是：

- 人能否一眼看清层级
- 续行是否明确
- 闭合关系是否清楚

### 4. 语言判断优先级

推荐顺序：

1. `pre` 上的 `codehub` 链接文件后缀
2. `pre.className`
3. `pre.hw-language`
4. 默认 `text`

例如：

- `.../Index.ets#...` -> `ArkTS`
- `.../Index.d.ts#...` -> `ts`
- `.../napi_init.cpp#...` -> `cpp`
- `language-plaintext` -> `text`

### 5. 示例源码链接可以保留在代码块后

如果代码块下方有源码链接，例如：

- `Index.ets`
- `napi_init.cpp`

可以保留为：

```md
[Index.ets](...)
```

这样既保留来源，又不污染代码块。

## 十二、Markdown 渲染稳定性规则

这部分是实践中补出来的硬规则，主要解决“网页内容看起来正常，但进 Obsidian 后排版错乱”的问题。

### 1. 普通文本中的尖括号必须转义

例如正文里出现：

- `<pid>`
- `<bundleName>`
- `<uri>`

如果直接写进 Markdown，Obsidian 会把它当成 HTML 标签片段，导致文本缺失或断句异常。

正确做法：

- `<pid>` -> `&lt;pid&gt;`
- `<bundleName>` -> `&lt;bundleName&gt;`

注意：

- 这条规则只针对普通正文、表格、列表、callout 内的文本
- 代码块内部不做这类转义，代码保持原样

### 2. callout、列表、段落混排时优先保证可渲染性

官网正文里经常出现：

- `说明` 块中嵌套列表
- 列表项里继续插入链接
- 一段话被 DOM 切成多段行内节点

如果机械按网页换行直接拼接，容易出现：

- 链接被拆到下一行
- 文本跑出列表层级
- callout 中断

处理原则：

- 先保证 Markdown 结构合法
- 再尽量贴近官网排版
- 同一列表项中的续行文本，需要保持为该列表项的延续内容
- callout 内每一行都必须继续带 `>` 前缀

### 3. 链接文本与前后正文必须按“行内内容”拼接

错误现象通常是：

- 前一句文字
- 链接单独断一行
- 后一句文字再断一行

这样在 Markdown 中常常会被误读为新段落，甚至破坏列表结构。

正确做法：

- 在同一段落、同一列表项、同一 callout 行内，链接应作为普通行内节点与前后文本合并输出
- 只有当原网页明确换段、换项、换标题时，才输出新的块级结构

### 5. 普通文本中的 `$` 符号必须转义

华为 HarmonyOS 文档中大量出现代码引用符号：

- `$r('app.type.name')` — 资源引用
- `$rawfile('filename')` — rawfile 资源引用
- `%1$s`、`%2$d` — 格式化字符串占位符

在 Markdown 中，`$` 会被 Obsidian 等渲染器解析为 LaTeX 数学公式分隔符。例如 `$r` 到下一个 `$` 之间的文本会被当作数学公式，导致显示异常或文本缺失。

正确做法：

- `$r(` → `\$r(`
- `$rawfile(` → `\$rawfile(`
- `%1$s` → `%1\$s`

注意：

- 代码块（` ``` `）内的 `$` 不转义（已被保护）
- 行内代码（`` ` ``）内的 `$` 不转义（已被保护）
- 只在普通正文、表格、列表、callout 内的文本中转义

### 6. 图片引用必须独立成行

HTML 转换后，经常出现图片引用与正文文本挤在同一行的情况：

```md
![](assets/doc/file-xxx.png)     **说明：** 正文内容...
```

这种格式在 Obsidian 中会导致图片无法正常渲染，且后续正文排版错乱。

正确做法：图片引用必须独立成行，与前后文本用换行分隔：

```md
![](assets/doc/file-xxx.png)

**说明：** 正文内容...
```

处理方式：

- 检测同时包含 `![](assets/...)` 和文本内容的行
- 将图片引用拆分为独立行
- 保留图片前后的文本为独立段落

### 7. 段落前导空格必须清除

HTML 标签转换后，大量的 DOM 缩进会残留为行首空格：

```md
     应用通过调用Device Security Kit的接口获取诈骗应用信息。
```

虽然 Markdown 渲染器通常会忽略段落前导空格，但大量空格影响源码可读性，且在某些场景下可能触发意外的代码块渲染。

正确做法：

- 去除所有行首的 2 个以上空格（`/^ {2,}\S/`）
- 例外：代码块内部、列表项（`- `、`1. `）、块引用（`> `）、表格行（`|`）保留原有缩进

### 8. 段落内多余空格必须压缩

HTML 中的 `&nbsp;` 以及 DOM 节点间的空白在转换后常变成大段连续空格：

```md
**流程说明：**                  用户在开发者应用上选择举报诈骗应用功能。
```

正确做法：

- 将 4 个以上连续空格压缩为单个空格（`\s{4,}` → ` `）
- 例外：代码块内部、表格行不处理

### 9. 图片路径中的空格必须 URL 编码

Obsidian 在解析 `![](assets/.../file.png)` 时，路径中的空格会导致图片无法渲染，显示为 `{data-source-line=N}` 的原始源码形式。

原因：标准 Markdown（CommonMark）规范中，`![]()` 的链接目标（link destination）如果是裸路径形式（不带 `<>`），不允许包含空格。Obsidian 的 Live Preview 严格遵循此规范。

错误示例：
```md
![](assets/ohos.app.ability.AppServiceExtensionAbility 应用后台服务扩展组件/file-xxx.png)
```

正确做法：
```md
![](assets/ohos.app.ability.AppServiceExtensionAbility%20应用后台服务扩展组件/file-xxx.png)
```

处理方式：
- 对 `![]()` 图片引用路径中的空格进行 URL 编码：` ` → `%20`
- 编码应在图片下载完成后、写入 Markdown 前执行
- 不影响文件系统路径（磁盘上目录名仍可用空格）
- 不影响代码块和行内代码

### 10. 不为了”贴网页 DOM”而破坏 Markdown 语法

网页 DOM 经常包含很多仅服务于前端渲染的包裹层：

- `div`
- `span`
- 高亮容器
- 表格展开容器
- 锚点容器

这些结构不应原样映射成 Markdown 层级。

转换时要遵循：

- 以语义块为准，不以 DOM 层级为准
- Markdown 输出必须是合法、稳定、可读的

### 11. API 元数据项之间必须有空行分隔（2026-05 新增）

API 参考文档的每个接口描述中，常出现连续的元数据行：

```md
**元服务API：** 从API version 12开始，该接口支持在元服务中使用。
**系统能力：** SystemCapability.ArkUI.ArkUI.Full
**模型约束：** 此接口仅可在Stage模型下使用。
**参数：**
| 参数名 | 类型 | 必填 | 说明 |
```

如果这些元数据行之间没有空行分隔，在 Obsidian 中会显得非常紧凑，可读性差。

正确做法——每项元数据之间保留空行：

```md
**元服务API：** 从API version 12开始，该接口支持在元服务中使用。

**系统能力：** SystemCapability.ArkUI.ArkUI.Full

**模型约束：** 此接口仅可在Stage模型下使用。

**参数：**

| 参数名 | 类型 | 必填 | 说明 |
```

处理规则：
- 连续的 `**xxx：**` 行之间必须插入空行
- `**参数：**` / `**返回值：**` 与紧随的表格之间需要空行
- `**示例：**` 与紧随的代码块之间维持现有空行即可
- 这条规则适用于所有 API 参考文档（harmonyos-references）

## 十三、HTML 转换中的常见陷阱（2026-05 新增）

以下是在大规模转换过程中发现的 HTML 结构陷阱，必须在转换器设计中预防。

### 1. `<p>` 正则不可用裸 `<p[^>]*>` —— 会误匹配 `<pre>`

HTML 中 `<p>` 和 `<pre>` 都以 `<p` 开头。正则 `<p[^>]*>` 会同时匹配 `<p align="...">` 和 `<pre class="...">`。

**修复**：使用词边界 `<p\b[^>]*>`。`\b` 确保 `<p` 后面不是字母，从而与 `<pre` 区分。

### 2. `<div>` 包裹层剥离必须先于内容处理

华为 API 返回的 HTML 中，表格、列表、代码块常被 `<div class="tablenoborder">`、`<div class="section">` 等包裹。如果先处理 `<p>` 标签再剥离 `<div>`，内部的 `<table>`、`<ul>` 等结构会被破坏。

**修复顺序**：
1. 提取 note callout（平衡 div 计数）
2. 剥离所有 `<div>` 标签（`\n` 替换）
3. 转换 `<pre>` → 代码块
4. 转换 `<table>` → Markdown 表格
5. 转换 `<h1>`-`<h4>` → 标题
6. 转换 `<img>` → 图片引用
7. 转换 `<ul>`/`<ol>` → 列表
8. 最后替换 `<p>` → 换行

### 3. `<div class="note">` 提取必须用 div 深度计数

Note 内部可能嵌套 `<div class="notebody">` 等子 div。用非贪婪正则 `([\s\S]*?)` 会在第一个 `</div>` 处截断，导致：
- 浅层截断：note 内容不完整
- 过度贪婪（无 `?`）：多段 note 之间的正文被吞掉，导致整节内容丢失

**修复**：从 `<div class="note"` 开始，对 `</div>` 做平衡计数，找到匹配的闭合标签。

### 4. 表格后处理：空列清除 + rowspan 补位

HTML 表格中常有空 `<td>` 和 `rowspan` 属性。转换后的 Markdown 需后处理：
- 删除所有单元格均为空的列（从右向左扫描）
- 将第一列的空单元格填入上一行的对应值（弥补 rowspan）

### 5. HTML 标签剥离必须在代码块保护之后

转换流程中多次出现 `replace(/<[^>]*>/g, '')` 以剥离残余 HTML 标签。但 `<pre>` 代码块在解码 `&lt;`/`&gt;` 后会包含 `<=`、`=>`、`<type>` 等含 `<>` 的文本。如果不清场保护就执行全局标签剥离，`<[^>]*>` 会从 `<=` 一直匹配到后续代码中的 `>`（可能跨越数百字符），删除中间所有内容。

**表现**：代码块不完整，混入其他代码块的片段。

**修复**：在每次全局 `replace(/<[^>]*>/g, '')` 之前，用 ` ```[\s\S]*?``` ` 保护所有 Markdown 代码块（占位符替换 → 剥离 → 恢复）。这在 `inline()` 函数和最终的 HTML 清理中均需执行。

### 6. 段落后处理：图片与文本拆分 + 空格压缩

转换流程结束后，必须对输出执行自动化后处理：
- 图片引用与同行文本拆分为独立行
- 行首 2+ 空格清除（保留列表/引用/表格/标题的缩进）
- 段落内 4+ 连续空格压缩为单空格

## 十四、为什么要整组重建，而不是只修单篇

当规则发生变化时，例如：

- 从 `'''ArkTS` 改为标准 fenced code block
- 从所有代码统一 `ArkTS` 改为按后缀识别语言
- 从 slug 文件名改为官方标题文件名

最稳的做法不是只修当前发现问题的一篇，而是整组重建。

原因：

- 同一模块内格式必须一致
- 否则后面会同时存在旧规则和新规则
- 返工成本反而更高

## 十四、转换后检查清单

每批 Markdown 完成后，至少检查：

1. 是否只保留正文页，没有 landing page
2. 文件名是否是官方文档名
3. 头部是否包含真实更新时间
4. 图片是否已经本地化
5. 图片路径是否符合 Obsidian 附件规则
6. 是否残留 HTML 表格
7. 代码块是否使用标准三反引号 fenced code block
8. 代码块语言是否正确
9. 是否混入 `收起 / 复制 / 自动换行 / 深色代码主题`
10. manifest 是否记录转换结果
11. 普通文本中的 `<pid>`、`<uri>` 这类占位符是否被正确转义
12. 是否还残留 `/consumer/...` 这种相对链接
13. callout、列表、段落中的链接是否被错误拆行
14. 代码块内部是否仍存在大段未缩进代码
15. JSON / JSON5 / 配置示例是否已经整理成标准层级
16. shell / 日志 / 工具输出是否被错误”美化”导致失真
17. 普通文本中的 `$r`、`$rawfile`、`%1$s` 等 `$` 符号是否已转义为 `\$`
18. 图片引用是否独立成行，没有和内文挤在同一行
19. 段落是否有残留的行首空格（2个以上）
20. 段落内是否有大量连续空格（4个以上未压缩）
21. 图片路径中的空格是否已 URL 编码为 `%20`
22. 表格是否有空列需要删除
23. 表格 rowspan 空单元格是否已填入上一行值
24. note callout 提取有无导致章节内容丢失
25. API 参考文档头部是否包含 `支持设备：` 信息（`device-type` 提取）
26. API 元数据项（`**元服务API：**`、`**系统能力：**`、`**模型约束：**`、`**参数：**`）之间是否有空行分隔

## 十五、推荐执行流程

完整推荐流程如下：

1. 读取 HTML 归档目录和源 manifest
2. 只筛选 `doc` 页
3. 读取每篇文档来源 URL
4. 打开官网实时页面
5. 提取：
   - `h1`
   - 更新时间
   - `.markdown-body`
   - 图片 URL
6. 下载图片到 `assets/${noteFileName}`
7. 清理无关 DOM
8. 转换标题、段落、列表、提示块
9. 转换表格为 Markdown
10. 转换代码块为标准 fenced code block，并识别语言
11. 写入 Markdown
12. 生成 Markdown 侧的 `manifest.json`
13. 抽检至少一篇含表格和代码块的文档
14. 删除临时脚本

## 十六、最终结论

把官网 HTML 整理成可长期维护的 Markdown，核心不是“转格式”，而是“定规范”。

真正稳定的方案必须同时满足：

- 目录结构稳定
- 文件命名稳定
- 更新时间真实
- 图片本地化
- 表格 Markdown 化
- 代码块语言正确
- 代码块语法标准
- Obsidian 附件路径稳定

如果这些规则不一次定清，后面一定会反复返工。

所以建议把 Markdown 整理阶段视为一个独立工序，而不是 HTML 抓取的附带动作。
